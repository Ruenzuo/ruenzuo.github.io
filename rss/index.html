]]>http://ruenzuo.github.io/ios-app-security/22a2189c-d5ef-46d1-8d1c-3f2db8f2be4bSat, 04 Oct 2014 15:03:49 GMT<![CDATA[Façade Software Design Pattern on iOS and Android]]>On this post I'll cover the façade software design pattern implemented on iOS and Android. This pattern has prove useful to me while developing engaging user experiences on mobile applications, why? Because mobile applications users expect apps to perform far better than web counterparts, and we can provide that experience by using system resources like disk (databases) and memory (caches) storages.

But if we don't structure all these system resources access methods in a comprehensive way we could risk ending up with a software design mess. Here's where the façade software design pattern comes into play, providing a single interface to solve a problem with different implementations.

To provide implementation examples, I'll be referencing these iOS and Android repositories. I'm not going to cover implementation details. I'll mostly cover the open source libraries I'm using and how I'm using them together, but if you're interested in implementation details feel free to leave me a message or send me an email to renzo.crisostomo@me.com.

Façade software design pattern explained

According to Wikipedia:

A facade is an object that provides a simplified interface to a larger body of code, such as a class library.

OK, that doesn't says much. So let's analyze a typical use case first.

Let's say we want to build an app that uses a REST API. We'll need to do network HTTP requests in order to retrieve data form that API, but we can't do this all the time because network requests are expensive, they consume battery and they take a lot of time depending on network conditions, so it make sense to cache the network request responses using all the provided mechanisms of the HTTP protocol. But what happens when the user doesn't have internet access? We could use databases and that would be fine, but disk I/O can be also expensive, depending of the database optimization that you could use, and if those requests will be done in a regular basis, it make sense to cache those too, we can use memory storage for that.

I guess that looks complex, we'll the truth is that not all the applications need to do all this, but if you're willing to use all the methods mentioned above in your application, you need to structure your classes in a comprehensive way. Take a look at this class diagram:

Here we have four classes, a manager and 3 helpers, implementing the same interface. Helpers are implementing the methods described above and the manager decides, depending several factors, which helper should be used to retrieve the requested data. Next, I'll talk a little about the libraries and techniques I've used to implement each helper both on iOS and Android.

Object serialization

Not pictured on the class diagram above for the sake of simplicity, but I'm using another helper: a translator helper. According to Wikipedia:

In computer science, in the context of data storage, serialization is the process of translating data structures or object state into a format that can be stored (for example, in a file or memory buffer, or transmitted across a network connection link) and reconstructed later in the same or another computer environment.

Object serialization has prove useful to me because it allows to manage any kind of data as objects. This objects are used as the input/output of the façade interfaces methods. Two powerful object serialization libraries I use are Mantle on iOS and Gson on Android. Both are able to translate JSON into model objects. You can find my translator helper implementation both on iOS and Android on the repositories mentioned earlier. These objects can implement other persistence strategies, like ActiveRecord, which I'll describe next.

Active record

According to Wikipedia:

Active record is an approach to accessing data in a database. A database table or view is wrapped into a class. Thus, an object instance is tied to a single row in the table. After creation of an object, a new row is added to the table upon save. Any object loaded gets its information from the database. When an object is updated the corresponding row in the table is also updated. The wrapper class implements accessor methods or properties for each column in the table or view.

Thanks to object serialization, now we have the data as objects, which can implement this pattern. Two powerful active record libraries I use are MagicalRecord on iOS and ActiveAndroid on Android. Implementing this pattern with both libraries is as easy as declare an inheritance or import a category, but I like to wrap the method calls on database helper. You can find my database helper implementation both on iOS and Android on the repositories mentioned earlier. Next, I'll cover network and memory caches.

Network and Memory caches

Enabling network cache is often as simple as declaring the size of it (really, see iOS and Android). Two powerful networking libraries that I use are AFNetworking on iOS and OkHttp on Android. Keep in mind that this will only work if the API that you're using is following the HTTP protocol cache methods. Memory cache on the other hard could be a little more tricky: you need to set up a key-value storage and specify a cache invalidation method. Two powerful memory cache libraries that I use are TMCache on iOS and Guava on Android. You can find my memory cache helper implementation both on iOS and Android on the repositories mentioned earlier.

Synchronizing large datasets

Techniques mentioned above, including the façade pattern haven't proved useful to me while synchronizing large datasets. I still haven't find any satisfying method to do this. I've seen some people downloading a pre-made database, but this can be tricky if your data varies a lot between users. What I'm doing now is running long background tasks and saving data on the database as often as I can, so the façade methods avoid network requests as much as possible. On iOS I'm using operations and on Android I'm using services. Be warned, code is really messy there, I'm not using any design pattern.

Closing

Things to keep in consideration while implementing something similar to this:

  • 250 countries data set took ~4 seconds to retrieve (serialized) from network, ~0.8 seconds from database and ~0.2 from memory (on an iPhone 4s and a Nexus 5), according to my own benchmarks which you can try yourself, among other things, on the repositories mentioned above.
  • Avoid network request as much as possible. Test your network cache methods.
  • Avoid cache a large amount of objects in memory and test your cache invalidation methods.
  • ActiveRecord and object serialization are two things difficult to get together on the same model. In my experience, using two different models on iOS is not such a bad idea, but probably there are better ways to do that.
  • All these is often over complex for most applications, it's up to you to decide when to use each of these strategies, or to use them all.

I'll hope this was helpful to you and please contact me if you have any inquiry about it.

Build system

We'll be using xctool as our build system for this. Advantages of this over the regular xcodebuild are well described by Mattt Thompson on NSHipster. Basically, we're gonna need compilation output in reports (namely junit and json-compilation-database formats) and xctool already includes these reporters.

So, to setup xctool with your project, you will need to install the script on your computer and add a .xctool-args file on your repository. Refer xctool documentation for information about this, but you'll need something like this.

Test results and Code coverage

You will need something like this on your CI server job in order to get test results:

xctool -reporter junit:Build/junit.xml clean test

This will run your test target and store the results on Build/junit.xml in the classic junit format. Most Jenkins CI installations already come with post-build actions for JUnit test result publishing, just setup one of those and you'll get something like this:

If you happen to run with this error on your CI server:

It means that you'll need to set your scheme to shared on Xcode. You can do this on Product > Schemes > Manage schemes.

Getting the code coverage is a little harder. First, you'll need to turn on two flags on your project debug main target: Generate Test Coverage Files and Instrument Program Flow. Then, you'll need to run something like this:

xctool clean test OBJROOT=./Build

This will store the test coverage files on a path I will describe on the next script, but first you need to install GCOVR (I always install those scripts on /opt/scripts/), a tool that will translate these results into Cobertura XML format. Then, you can get the cobertura results with this:

See that we're looking inside Debug-iphonesimulator, the target where we enable the two flags. On Jenkins CI, you will need to install Cobertura plugin, setup a post-build action and publish those results. You'll get something like this:

If you're getting blank results, at the time of this writing, there's a bug with Xcode 5. In order to get pass through this, you will need to call __gcov_flush() from your application using a test hook, details on how to do this are shared on this StackOverflow question.

PMD analysis results

First, install OCLint. This tool will need a json-compilation-database which is just the list of commands that xcodebuild runs. You can get those with xctool using the following:

xctool -reporter json-compilation-database:compile_commands.json clean build

Then, run OCLint:

oclint-json-compilation-database -e Pods/** -- -max-priority-1 20 -max-priority-2 50 -max-priority-3 100 -report-type pmd -o Build/oclint.xml

There we're excluding Pods directory, because we're not interested on those sources. Max priority values are set in order to make the build fail if those are exceeded.

Probably you're already familiar with this concept, if not, this is an analysis performed on the source code in order to get its quality metrics. I'll be diving this in two parts. On the first I'll cover the metrics I use, briefly describing each one of them. The quality metrics described below are:

  • Test results and Code coverage
  • PMD analysis results
  • Duplicate code trends
  • Source lines of code count

Test results and Code coverage

Unit testing is a fundamental part of the software development process. As described on Mike Cohn's book "Succeeding with Agile", testing automation efforts on software development should be distributed as shown below:


This means that "unit testing should be the foundation of a solid test automation strategy and as such represents the largest part of the pyramid".

This is why this two metrics are are important to me, test results helps you to identify which builds have been broken and coverage gives you interesting information about how much of your code is being covered by your unit tests.

PMD analysis results

PMD is a source code analyzer. It finds common programming flaws like unused variables, empty catch blocks, unnecessary object creation, and so forth. Currently, PMD doesn't have official support for ObjC, but on the second part of this post I'll cover how to get through this.

I found this metrics interesting because it helps me to write clean code avoiding common errors like long non-commenting source statements, long lines, switch statements without default, unnecessary else statements, unused method parameters and so on.

Duplicate code trends

This metric shows you code duplication tagging two sections of the source file. It compares them with a set of parameters you specify to give you the results. This helps you with the DRY software development principle.

There's nothing more to say about this metric besides that is possible the hardest to configure of all, mostly because the parameters you specify to the program depends on the coding style of the frameworks you use and your team, which may not be the same at all.

Source lines of code count

I believe that source lines of code count by itself doesn't really mean anything. You can't measure the size or complexity of a software program only by lines of code count, but this metric is really helpful if it's used with the other described above, specially with duplicate code trends. This is possibly the hardest to analyze of all, because it depends mostly on the complexity of the project and the team.

If you found this interesting, keep reading the second part of this to learn how to configure these quality metrics.

I'm assuming that you are familiar with those tools, so I'll be covering all these in the following topics:

What is continuous delivery?

Continuous delivery is a design practice used in software development to automate and improve the process of software delivery. Yes, that's Wikipedia.

Probably you're familiar with this:

That is Scrum, but I will not explain this here, we will only focus on the last two graphs: 2-4 week sprint and potentially shippable product increment. A sprint is an iteration, very similar to what we find on other agile software development methodologies and the goal of the sprint execution is a potentially shippable product increment. But, two weeks seems like a really short period of time for doing requirements, design, construction and release tasks for a set of features, so in order to get the most of it, we do automation.

Motivations and goals

Working on mobile projects means that probably on of your main goals is time to market, because let's face it: mobile application development is probably one of the most competitive environments on software right now. So, in order to arrive earlier to the party, you need to decide which features will be on your first release.

One interesting approach to do this, and this one is actually being used by Facebook, is to have multiple version of the same application each one with different features. These versions are often distributed to different quality assurance teams. You think this looks complicated? Well, it's not!

Finally, these will be our goals:

  • Build different versions of an Android application using the same code base.
  • Each version should have a determined set of features and those features should be tested individually with instrumentation framework.

  • eXtreme Programming

On the second, you have a set of practices and some of them are really hard to get done right, like: test driven development and coding standards are really advanced stuff for junior software developers, continuous integration and small releases are somehow difficult to accomplish depending on your IT infrastructure, collective ownership could be a nightmare done wrong.

  • Tapbots, creators of the amazing iOS app Tweetbot.
  • Team Meat, creators of the video game hit Super Meat Boy. Most tasks on software development and design must be reviewed by different points of view, and it's so easy to miss a thing... If you want to learn more on how this teams are organized, I recommend reading both teams blogs and watch this awesome movie.

And reading both Tapbots and Team Meat blogs got me an idea on how are they organized, which is again very similar to scrum.

Scrum isn't recommended for teams of two people and I couldn't find information regarding an agile methodology for teams conformed by two people, so there's an interesting field of research there.

How are these three thoughts related?

  • Independent software development it's hard. You have to fulfill multi-disciplinary roles and you have to be the best on each one, pretty much like doing agile, the right way.
  • You can't do everything on your own, but it seems that there is a kind of magic formula with two people teams.
  • There's an interesting field of research on this two people independent software development teams using kind of agile practices.
