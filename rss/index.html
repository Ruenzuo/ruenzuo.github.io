For this part, I'll be referencing this repository. If you're interested in what are those metrics or what are the motivations behind this, be sure to check the first part of this post.

The following assumes that you have basic knowledge about Jenkins CI and Linux Shell scripting.

Build system

We'll be using xctool as our build system for this. Advantages of this over the regular xcodebuild are well described by Mattt Thompson on NSHipster. Basically, we're gonna need compilation output in reports (namely junit and json-compilation-database formats) and xctool already includes these reporters.

So, to setup xctool with your project, you will need to install the script on your computer and add a .xctool-args file on your repository. Refer xctool documentation for information about this, but you'll need something like this.

Test results and Code coverage

You will need something like this on your CI server job in order to get test results:

xctool -reporter junit:Build/junit.xml clean test

This will run your test target and store the results on Build/junit.xml in the classic junit format. Most Jenkins CI installations already come with post-build actions for JUnit test result publishing, just setup one of those and you'll get something like this:

If you happen to run with this error on your CI server:

ERROR: Cannot find schemes. Please consider creating shared schemes in Xcode.

It means that you'll need to set your scheme to shared on Xcode. You can do this on Product > Schemes > Manage schemes.

Getting the code coverage is a little harder. First, you'll need to turn on two flags on your project debug main target: Generate Test Coverage Files and Instrument Program Flow. Then, you'll need to run something like this:

xctool clean test OBJROOT=./Build

This will store the test coverage files on a path I will describe on the next script, but first you need to install GCOVR (I always install those scripts on /opt/scripts/), a tool that will translate these results into Cobertura XML format. Then, you can get the cobertura results with this:

/opt/scripts/gcovr -r . --object-directory Build/TestingApp.build/Debug-iphonesimulator/WeatherApp.build/Objects-normal/i386 --exclude '.*Tests.*' --exclude '.*main.*' --xml > Build/coverage.xml

See that we're looking inside Debug-iphonesimulator, the target where we enable the two flags. On Jenkins CI, you will need to install Cobertura plugin, setup a post-build action and publish those results. You'll get something like this:

If you're getting blank results, at the time of this writing, there's a bug with Xcode 5. In order to get pass through this, you will need to call __gcov_flush() from your application using a test hook, details on how to do this are shared on this StackOverflow question.

PMD analysis results

First, install OCLint. This tool will need a json-compilation-database which is just the list of commands that xcodebuild runs. You can get those with xctool using the following:

xctool -reporter json-compilation-database:compile_commands.json clean build

Then, run OCLint:

oclint-json-compilation-database -e Pods/** -- -max-priority-1 20 -max-priority-2 50 -max-priority-3 100 -report-type pmd -o Build/oclint.xml

There we're excluding Pods directory, because we're not interested on those sources. Max priority values are set in order to make the build fail if those are exceeded.

In order to publish those results on your Jenkins CI, you'll need to install PMD plugin, setup a post-build action and publish those results. Then you'll get something like this:

Duplicate code trends

This one is fairly straightforward. You'll need to download PMD and Objective-C-CPD-Language (PMD version 4.2.5 and Objective-C-CPD-Language version 0.0.7 worked for me). Then run something like this:

java -Xmx512m -classpath /Applications/pmd/lib/pmd-4.2.5.jar:/Applications/pmd/lang/ObjCLanguage-0.0.7-SNAPSHOT.jar net.sourceforge.pmd.cpd.CPD --minimum-tokens 10 --language ObjectiveC --encoding UTF-8 --format net.sourceforge.pmd.cpd.XMLRenderer --files WeatherApp --files WeatherAppTests > Build/cpd-output.xml

In order to publish those results on your Jenkins CI, you'll need to install DRY plugin, setup a post-build action and publish those results. Then you'll get something like this:

Source lines of code count

First, install CLOC. Then run something like this:

cloc --by-file --xml -out=Build/cloc.xml WeatherApp

This will store the source lines of code count on a xml file but in order to get those published by Jenkins CI, we'll need to translate that xml to SLOCCount format with this:

xsltproc Utils/Sloccount-format.xls Build/cloc.xml > Build/cloccount.sc

xsltproc already comes preinstalled on OSX and you can grab an example of Sloccount-format.xls here. The reason why I'm doing this is because I couldn't get SLOCCount work on OSX Mavericks for some reason, so if you can solve this, please let me know.

Probably you're already familiar with this concept, if not, this is an analysis performed on the source code in order to get its quality metrics. Remember: "You can't improve what you don't measure".

I'll be diving this in two parts. On the first I'll cover the metrics I use, briefly describing each one of them. On the second part I'll show you how to configure these quality metrics on a dashboard using Jenkins continuous integration server. For this last part, I'll be referencing this repository a lot.

The quality metrics described below are:

  • Test results and Code coverage
  • PMD analysis results
  • Duplicate code trends
  • Source lines of code count

Test results and Code coverage

Unit testing is a fundamental part of the software development process. As described on Mike Cohn's book "Succeeding with Agile", testing automation efforts on software development should be distributed as shown below:


This means that "unit testing should be the foundation of a solid test automation strategy and as such represents the largest part of the pyramid".

This is why this two metrics are are important to me, test results helps you to identify which builds have been broken and coverage gives you interesting information about how much of your code is being covered by your unit tests.

PMD analysis results

PMD is a source code analyzer. It finds common programming flaws like unused variables, empty catch blocks, unnecessary object creation, and so forth. Currently, PMD doesn't have official support for ObjC, but on the second part of this post I'll cover how to get through this.

I found this metrics interesting because it helps me to write clean code avoiding common errors like long non-commenting source statements, long lines, switch statements without default, unnecessary else statements, unused method parameters and so on.

Duplicate code trends

This metric shows you code duplication tagging two sections of the source file. It compares them with a set of parameters you specify to give you the results. This helps you with the DRY software development principle.

There's nothing more to say about this metric besides that is possible the hardest to configure of all, mostly because the parameters you specify to the program depends on the coding style of the frameworks you use and your team, which may not be the same at all.

Source lines of code count

I believe that source lines of code count by itself doesn't really mean anything. You can't measure the size or complexity of a software program only by lines of code count, but this metric is really helpful if it's used with the other described above, specially with duplicate code trends. This is possibly the hardest to analyze of all, because it depends mostly on the complexity of the project and the team.

If you found this interesting, keep reading the second part of this to learn how to configure these quality metrics.

I'm dividing this in two parts because of course divide and conquer.

What is continuous delivery?

Continuous delivery is a design practice used in software development to automate and improve the process of software delivery. Yes, that's Wikipedia.

Probably you're familiar with this:

That is Scrum, but I will not explain this here, we will only focus on the last two graphs: 2-4 week sprint and potentially shippable product increment. A sprint is an iteration, very similar to what we find on other agile software development methodologies and the goal of the sprint execution is a potentially shippable product increment. But, two weeks seems like a really short period of time for doing requirements, design, construction and release tasks for a set of features, so in order to get the most of it, we do automation.

Motivations and goals

Working on mobile projects means that probably on of your main goals is time to market, because let's face it: mobile application development is probably one of the most competitive environments on software right now. So, in order to arrive earlier to the party, you need to decide which features will be on your first release.

One interesting approach to do this, and this one is actually being used by Facebook, is to have multiple version of the same application each one with different features. These versions are often distributed to different quality assurance teams. You think this looks complicated? Well, it's not!

Finally, these will be our goals:

  • Build different versions of an Android application using the same code base.
  • Each version should have a determined set of features and those features should be tested individually with instrumentation framework.
  • Use different package identifiers and application icons so we can have multiple versions of the application installed and tested on the same devices.
  • Distribute those versions individually on a continuous delivery environment.
  • Trigger distributions based on commits on a specific branch.
  • Each version should be distributed to a different set of recipients.

I guess that looks interesting, right? Keep reading the second part of this if you want to learn how to do all these.

  • What they don't tell you about Agile... Or do they?
  • You can't do everything on your own... Or can you?
  • What's the best way to do everything on your own? You may be wondering how they are related... Let me explain myself.

What they don't tell you about Agile

Agile is hard. Yes, we all have been lied about it. Let's analyze for instance two agile methodologies:

  • Scrum
  • eXtreme Programming

On the first, you have three main roles: scrum master, product owner and the team, let's focus on the last one. As a team member, you should be involved on several tasks performing multiple roles in different disciplines such as requirements, design, construction, testing, maintenance and configuration management. Specialization is not forbidden, but multi-disciplinary teams are encourage.

On the second, you have a set of practices and some of them are really hard to get done right, like: test driven development and coding standards are really advanced stuff for junior software developers, continuous integration and small releases are somehow difficult to accomplish depending on your IT infrastructure, collective ownership could be a nightmare done wrong.

You get the idea.

You can't do everything on your own

You can't. I think some of us are getting the wrong idea about indie software development... What's indie on these days anyways? I'll explain this with two successful independent software development teams:

  • Tapbots, creators of the amazing iOS app Tweetbot.
  • Team Meat, creators of the video game hit Super Meat Boy. Both teams are conformed by a developer and a designer, so it's clear that those aren't lone wolves and I think there's a reason about that. Most tasks on software development and design must be reviewed by different points of view, and it's so easy to miss a thing... Also, development and testing tasks should be done by someone different, otherwise is a disaster. If you want to learn more on how this teams are organized, I recommend reading both teams blogs and watch this awesome movie.

What's the best way to do everything on your own

Is Agile the best way? I think it is the right direction. I have been told that Apple apps teams are conformed by 3 people: a developer, a designer and a marketing analyst working with a methodology very similar to scrum. And reading both Tapbots and Team Meat blogs got me an idea on how are they organized, which is again very similar to scrum.

Scrum isn't recommended for teams of two people and I couldn't find information regarding an agile methodology for teams conformed by two people, so there's an interesting field of research there.

How are these three thoughts related?

  • Independent software development it's hard. You have to fulfill multi-disciplinary roles and you have to be the best on each one, pretty much like doing agile, the right way.
  • You can't do everything on your own, but it seems that there is a kind of magic formula with two people teams.
  • There's an interesting field of research on this two people independent software development teams using kind of agile practices.

All these has my attention for further research, so I'll keep you posted on this when I'll have more information.
